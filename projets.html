Stage 2ème année ⭐ Oral E6

Stage — Déploiement de postes (FOG Project)

Lycée Sidoine Apollinaire, Clermont-Ferrand — 23 fév. au 3 avr. 2026 — Binôme : Yanis Noël — Tuteur : M. Rochon

Contexte

Stage de 6 semaines au Lycée Sidoine Apollinaire. Mission : déployer une image Windows 11 standardisée sur ~60 postes dans 3 salles (S-402, S-405, S-406) via un serveur FOG Project (clonage réseau PXE/Multicast). Machines : PC TERRA (DDR5, 1 To). Travail en totale autonomie.

Architecture réseau

Réseau dédié — 192.168.10.0/24 Serveur FOG 192.168.10.12 — VM Debian 11, déploiement PXE/Multicast Windows Server 192.168.10.20 — VM, DHCP (opt. 066/067 PXE) Routeur Cisco 192.168.10.254 — Passerelle Plage DHCP 192.168.10.40–100 — Postes clients PXE

Logiciels dans l'image master

Réseau
Cisco Packet Tracer · Wireshark
Développement
Visual Studio 2026 · VS Code · Notepad++
Virtualisation
VMware Workstation · VirtualBox
Web local
WAMP + PHP
Utilitaires
Chrome · CCleaner

Phases du projet

Phase 1

Analyse & planification (GANTT)

▾
Inventaire matériel (PC TERRA, switch + routeur Cisco), choix de FOG Project v1.5.10, élaboration d'un diagramme de Gantt sur 6 semaines. Planification salle par salle selon les créneaux de disponibilité.
Phase 2

Infrastructure réseau

▾
  • Installation du serveur FOG (VM Debian 11, IP statique 192.168.10.12).
  • Config DHCP Windows Server : option 066 → 192.168.10.12, option 067 → ipxe.efi.
  • Configuration switch + routeur Cisco pour le réseau de déploiement.
Phase 3

PC Master + Sysprep

▾
Installation Windows 11 + logiciels sur le PC Master. Généralisation via Sysprep (OOBE + « Généraliser ») pour réinitialiser le SID et éviter les conflits réseau/AD. PC éteint immédiatement, jamais redémarré sur Windows avant capture.
Phase 4

Capture (~70 Go) + tests Unicast

▾
Capture via boot PXE (Quick Registration + tâche FOG). Image ~70 Go. Tests Unicast sur 1-2 postes : validation drivers, logiciels, nommage.
Phase 5

Déploiement Multicast (~60 postes)

▾
  • Config BIOS : boot UEFI LAN prioritaire, Network Stack IPv4, Secure Boot désactivé.
  • Enregistrement postes via Quick Registration and Inventory FOG.
  • Déploiement Multicast salle par salle (~20 postes simultanément).
  • Post-déploiement : intégration au domaine Active Directory du lycée.
Phase 6

Tâches complémentaires

▾
  • Veyon : supervision des postes élèves (contrôle distant, partage écran, verrouillage).
  • Zabbix : supervision infrastructure réseau du lycée (alertes, monitoring services).

Compétences

C1 — Gérer le patrimoine informatique C4 — Travailler en mode projet C5 — Mettre à disposition un service C6 — Développement professionnel

TP ⭐ Oral E6

GLPI + OCS Inventory NG — Gestion de parc

Lycée Sidoine Apollinaire, Clermont-Ferrand — Janvier 2026 — Binôme : Joshua

Contexte

Mission : concevoir et déployer une infrastructure de gestion de parc informatique pour la section BTS SIO du lycée, à l'aide de GLPI couplé à OCS Inventory NG pour l'inventaire automatisé.

Architecture technique

Serveur Debian (srv-bts-sio) Apache2 + PHP + MariaDB OCS Inventory NG — Inventaire automatisé (agent → serveur) GLPI 10.0.14 — Gestion parc + tickets d'incidents Plugin OCSNG 2.0.5 — Interconnexion OCS ↔ GLPI Entités : BTS SIO > Salle B-405 / B-406 / B-407

Organisation mise en place

Entités

Structure organisationnelle

▾
Entité racine BTS SIO avec sous-entités : Salle B-405, B-406, B-407. Permet de localiser chaque incident précisément et de cibler les interventions.
Profils

Gestion des droits utilisateurs

▾
  • Étudiants : création et suivi de leurs propres tickets uniquement.
  • Enseignants : vue globale sur tous les tickets de leur entité.
  • Administrateurs/Techniciens : gestion complète du parc et des tickets.
OCS

Inventaire automatisé

▾
Agents OCS sur les postes Windows → remontée automatique des infos matérielles (CPU, RAM, disque) et logicielles (OS, licences, domaine). Chaque équipement est lié à son historique de maintenance dans GLPI.
Tickets

Cycle de vie des incidents

▾
Cycle : Création → Attribution → En Attente → Résolution → Clôture
  • Cas 1 : "PC ne démarre pas" (Salle B-407) — ouverture étudiant, résolution technicien, clôture documentée.
  • Cas 2 : Visibilité selon profil — étudiant voit uniquement son ticket, enseignant voit tous les tickets de son entité.

Compétences

C1 — Gérer le patrimoine informatique C2 — Répondre aux incidents C5 — Mettre à disposition un service C6 — Développement professionnel

GSB — HAProxy + Nagios

GSB-DIJON-2, Lycée Sidoine Apollinaire — Nov.–Déc. 2025 — Binôme : Yanis Noël — N° candidat : 2602989488

Contexte

Dans le cadre de l'AP GSB (agence GSB-DIJON-2), deux missions font l'objet des fiches E6 officielles : HAProxy (Fiche n°1) et Nagios (Fiche n°2), déployées en zone DMZ de l'infrastructure réseau de l'agence.

Infrastructure GSB-DIJON-2

LAN — VLAN 1203 — 192.168.227.96/28 WINSRVAD-DIJON 192.168.227.97 — AD DS / DHCP / DNS / w32tm Win10 (PC1,2,3) 192.168.227.101–104 — DHCP DMZ — VLAN 1211 — 10.31.2.0/24 SRVWEBSSH 10.69.2.1 — Apache2 + SSH SRVWEBSSH_2 10.69.2.2 — Apache2 SRVHAPROXY 10.69.2.3 — HAProxy (Round Robin 50/50) SRVNAGIOS 10.69.2.4 — Supervision (CPU/RAM/disk/HTTP) IPFIRE (DMZ) 10.69.2.254 vSphere : Datacenter_SIO63\E2\AP CERTA NOEL

Fiche E6 n°1 — HAProxy (03/11/2025 – 18/11/2025)

  • Installation de HAProxy sur VM Debian en DMZ (10.69.2.3).
  • Frontend : écoute port 80 sur l'IP du HAProxy.
  • Backend : algorithme roundrobin 50/50 entre SRVWEBSSH et SRVWEBSSH_2.
  • DNS Dnsmasq : résolution gsb-dijon-2-web.com → 10.69.2.3.
  • Tests : alternance visible entre les deux serveurs à chaque requête HTTP.
  • Config avancée : simulation serveur 3× moins puissant (pondération roundrobin).

Fiche E6 n°2 — Nagios (01/12/2025 – 16/12/2025)

  • Installation de Nagios sur VM Debian en DMZ (10.69.2.4).
  • Protocoles : NRPE (Linux), NSClient++ (Windows), ICMP (ping).
  • Éléments supervisés : charge CPU, RAM, espace disque, service HTTP.
  • Cibles : SRVWEBSSH, SRVWEBSSH_2, SRVHAPROXY, WINSRVAD-DIJON.
  • Alertes temps réel accessibles depuis l'interface web (LAN admin).

Compétences
