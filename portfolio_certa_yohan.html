Stage 2ème année⭐ Sélectionné pour l'oral E6
Stage — Déploiement de postes (FOG Project)
Lycée Sidoine Apollinaire, Clermont-Ferrand — 23 fév. au 3 avr. 2026 — Binôme : Yanis Noël — Tuteur : M. Rochon

Contexte

Stage de 6 semaines au Lycée Sidoine Apollinaire (23 février – 3 avril 2026), en binôme avec Yanis Noël, sous la supervision de M. Rochon. Mission principale : déployer une image Windows 11 standardisée sur ~60 postes dans 3 salles informatiques (S-402, S-405, S-406) via un serveur FOG Project (Free Open-source Ghost, clonage réseau via PXE). Les PC sont des machines TERRA (DDR5, 1 To). Travail en totale autonomie hors interventions ponctuelles du tuteur.

Architecture de déploiement

Infrastructure réseau dédiée — 192.168.10.0/24 Serveur FOG 192.168.10.12 — VM Debian 11, déploiement PXE/Multicast Windows Server 192.168.10.20 — VM, serveur DHCP (opt. 066/067 PXE) Routeur Cisco 192.168.10.254 — Passerelle Plage DHCP 192.168.10.40–100 — Postes clients (boot PXE)

Réseau

Phases du projet

Phase 1

Analyse & planification (GANTT)

▾
Inventaire du matériel (PC TERRA, switch + routeur Cisco), choix de FOG Project v1.5.10, élaboration d'un diagramme de Gantt sur 6 semaines. Planification salle par salle selon les créneaux de disponibilité (cours en cours). Les PC étaient temporairement stockés au gymnase avant remontée au 4e étage.
Phase 2

Mise en place de l'infrastructure réseau

▾
  • Installation du serveur FOG (VM Debian 11, IP statique 192.168.10.12) via le script officiel FOG v1.5.10.
  • Configuration du serveur DHCP Windows Server : option 066 → 192.168.10.12 (boot server), option 067 → ipxe.efi (UEFI PXE).
  • Configuration du switch + routeur Cisco pour le réseau 192.168.10.0/24.
  • Refus du DHCP intégré FOG (délégué au Windows Server), HTTPS décliné pour simplifier la mise en place.
Phase 3

Préparation du PC Master + Sysprep

▾
Installation propre de Windows 11 + tous les logiciels requis sur le PC Master. Généralisation via Sysprep (mode OOBE + « Généraliser ») pour réinitialiser le SID et éviter les conflits réseau/AD lors du déploiement sur les 60 machines. PC éteint immédiatement après Sysprep, sans redémarrage Windows.
Phase 4

Capture de l'image (~70 Go) + tests Unicast

▾
Création d'une tâche de capture FOG via Quick Registration. Boot PXE du Master → FOG aspire l'intégralité du disque (~70 Go) vers le serveur. Tests en Unicast sur 1-2 postes : validation des drivers (son, réseau, affichage), des logiciels et du nommage avant déploiement massif.
Phase 5

Déploiement Multicast des 3 salles (~60 postes)

▾
  • Configuration BIOS de chaque PC : boot UEFI LAN en priorité, Network Stack IPv4, désactivation du Secure Boot.
  • Enregistrement de chaque poste via Quick Registration and Inventory de FOG.
  • Déploiement en Multicast salle par salle (~20 postes simultanément).
  • Post-déploiement : intégration de tous les postes au domaine Active Directory du lycée.
Phase 6

Tâches complémentaires

▾
  • Veyon : supervision des postes élèves (contrôle à distance, partage d'écran, verrouillage des postes).
  • Zabbix : supervision de l'infrastructure réseau du lycée (alertes, monitoring des services).

Difficultés rencontrées & solutions

  • Installation FOG : problème de configuration réseau → résolu en ajustant les paramètres d'interface.
  • Erreur Sysprep : packages Windows en conflit → désinstallation des applications concernées avant Sysprep.
  • Multicast FOG : timeout de session trop court → ajustement de la configuration FOG.
  • Contrainte matérielle : certains PC sans sortie VGA compatible avec les écrans disponibles → adaptation de la configuration.

GLPI + OCS Inventory NG — Gestion de parc
Lycée Sidoine Apollinaire, Clermont-Ferrand — Janvier 2026 — Binôme : Joshua

Contexte

Mission de stage au Lycée Sidoine Apollinaire consistant à concevoir et déployer une infrastructure de gestion de parc informatique pour la section BTS SIO, à l'aide de GLPI (Gestion Libre de Parc Informatique) couplé à OCS Inventory NG pour l'inventaire automatisé.

Architecture technique

Serveur Debian (srv-bts-sio) Apache2 + PHP + MariaDB OCS Inventory NG — Inventaire automatisé des postes GLPI 10.0.14 — Gestion parc + tickets d'incidents Plugin OCSNG 2.0.5 — Interconnexion OCS ↔ GLPI Client Windows 10 Agent OCS Inventory — Remontée automatique des infos matérielles/logicielles

Organisation GLPI mise en place

Entités

Structure organisationnelle

▾
Entité racine BTS SIO avec sous-entités correspondant aux salles physiques : Salle B-405, Salle B-406, Salle B-407. Cette hiérarchie permet de localiser précisément chaque incident et de cibler les interventions.
Profils

Gestion des droits utilisateurs

▾
  • Étudiants (profil Utilisateur) : création et suivi de leurs propres tickets uniquement.
  • Enseignants (profil Observateur/Technicien) : vue globale sur les tickets de leur entité, priorisation des interventions.
  • Administrateurs/Techniciens : gestion complète du parc, attribution et résolution des tickets.
Inventaire

OCS Inventory NG — remontée automatique

▾
Couplage GLPI + OCS Inventory NG : les agents OCS installés sur les postes Windows remontent automatiquement les informations matérielles (processeur, RAM, disque) et logicielles (OS, licences, domaine). Chaque équipement est directement lié à son historique de maintenance dans GLPI.
Tickets

Cycle de vie des incidents

▾
Cycle en 5 étapes : Création → Attribution → En Attente → Résolution → Clôture
  • Cas 1 : "PC ne démarre pas" (Salle B-407) — ouverture par un étudiant, prise en charge technicien, résolution (rebranchement câble alimentation), clôture avec impact et urgence documentés.
  • Cas 2 : Démonstration de la visibilité selon les droits — l'étudiant ne voit que son ticket, l'enseignant voit tous les tickets de son entité.

Installation technique réalisée

  • Installation de la stack LAMP sur Debian 12 (Apache2, MariaDB, PHP 8.2 + modules).
  • Installation et configuration d'OCS Inventory NG : base de données ocsweb, utilisateur ocs, dépôt officiel Debian, modules Perl CPAN.
  • Installation et configuration de GLPI 10.0.14 : base de données glpidb, déploiement via interface web.
  • Installation du plugin OCSNG 2.0.5 pour interconnecter OCS et GLPI.
  • Déploiement de l'agent OCS sur un client Windows 10 et vérification de la remontée d'inventaire.

Contexte

Dans le cadre de l'AP GSB (agence GSB-DIJON-2, Dijonnaise), deux missions ont fait l'objet de fiches E6 officielles : la mise en place d'un répartiteur de charge HAProxy (Fiche n°1) et d'une solution de supervision Nagios (Fiche n°2), toutes deux déployées en zone DMZ de l'infrastructure réseau de l'agence.

Fiche E6 n°1 — HAProxy (03/11/2025 – 18/11/2025)

Objectif : Déployer un répartiteur de charge en DMZ pour distribuer le trafic HTTP entre deux serveurs web de manière équitable (haute disponibilité).

  • Installation de HAProxy sur une VM Debian en DMZ (SRVHAPROXY, 10.69.2.3).
  • Configuration du frontend : écoute sur l'IP du HAProxy, port 80.
  • Configuration du backend : algorithme roundrobin 50/50 entre SRVWEBSSH et SRVWEBSSH_2.
  • Configuration DNS Dnsmasq : résolution gsb-dijon-2-web.com → 10.69.2.3.
  • Tests de vérification : pages web différentes sur chaque serveur, alternance visible à chaque requête.
  • Configuration avancée : simulation d'un serveur 3× moins puissant (pondération roundrobin).

Fiche E6 n°2 — Nagios (01/12/2025 – 16/12/2025)

Objectif : Déployer une solution de supervision pour surveiller l'état de santé de l'ensemble des équipements de l'agence.

  • Installation de Nagios sur une VM Debian en DMZ (SRVNAGIOS, 10.69.2.4).
  • Protocoles utilisés : NRPE (agents Linux), NSClient++ (Windows Server), ICMP (ping).
  • Éléments supervisés : charge CPU, utilisation RAM, espace disque, service HTTP (web).
  • Cibles : SRVWEBSSH, SRVWEBSSH_2, SRVHAPROXY, WINSRVAD-DIJON.
  • Alertes en temps réel accessibles depuis l'interface web via le LAN administrateur.
